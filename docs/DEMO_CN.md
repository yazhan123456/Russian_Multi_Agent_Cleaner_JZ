# 中文 Demo 讲解稿

这份文档用于中文面试或中文项目展示。

## 一句话介绍

这是一个面向俄语学术、法律、历史 PDF 的文档清洗系统。它不是简单做一次 OCR，而是先做版面过滤，再做 OCR 或文本提取路由，然后做清洗、修复、结构恢复和导出后规则收尾，最后输出更干净的纯文本语料。

## 这个项目解决什么问题

普通 OCR 处理俄语长文档时，经常会遇到这些问题：

- 页眉页码混进正文
- 图注、尾注、参考文献污染正文
- 图片和表格区域被当成正文处理
- A3 双页扫描导致版面分析混乱
- OCR 产生断词、连字符断裂、词语粘连
- 长文档处理中断后难以恢复

这个项目的目标不是“识别文字”本身，而是把一整本脏 PDF 变成更适合研究、检索、RAG 使用的文本。

## 30 秒版本

如果面试官问“这个项目是什么”，可以这样说：

> 我做的是一个俄语长文档清洗系统，目标是把学术和法律 PDF 变成更干净的纯文本语料。它先做版面分流和局部遮罩，再决定是直接抽取文本还是走 OCR，后面再做清洗、修复和结构恢复。这个系统支持页级 checkpoint 和 resume，适合处理整本书，而不是只做几页 OCR 演示。

## 2 分钟版本

可以按这个结构讲：

1. 先说问题
   - 俄语 PDF 不只是 OCR 问题，还涉及图注、尾注、页眉、断词、双页扫描等噪声。

2. 解释第一步
   - 我用 Paddle 做版面分流。
   - 正文和标题保留，图、表、注释区域遮掉。

3. 解释第二步
   - 后面不是所有页都强制走 OCR。
   - 如果 PDF 文本层足够好，就直接 extract。
   - 风险页才走 OCR。

4. 解释后处理
   - 规则清洗去掉明显噪声。
   - DeepSeek 负责清洗、修复和结构恢复。
   - 导出后还有一层规则收尾，专门处理图注、断词、同形字、页眉残留等问题。

5. 解释工程点
   - 页级状态机
   - checkpoint / resume
   - A3 双页拆分
   - 导出后规则清洗

## 架构要点

这不是一个“单模型自由发挥”的 Agent，而是一个工程化 pipeline：

1. 文档预处理
2. Paddle layout sanitize
3. OCR / extract 路由
4. Rule cleaning
5. Primary cleaning
6. Review
7. Repair
8. Structure restore
9. Export + post-clean

系统重点不是“某个模型多强”，而是：

- 任务拆分是否清晰
- 状态是否可恢复
- 成本和速度是否可控
- 脏页和好页是否能区别处理

## 可以强调的工程亮点

- 支持长文档整本处理，而不是只做几页 demo
- 页级 checkpoint 和 resume，处理中断不会整本重来
- A3 双页会先拆分，再做版面分析
- Paddle 先做遮罩，减少人工裁图、裁尾注
- `extract` 与 OCR 路由分流，避免无意义的重 OCR
- 导出后规则可以修图注、断词、词语粘连、页眉残留

## 面试官可能会问什么

### 为什么不用一个多模态大模型一次做完？

因为这里的问题不只是 OCR，而是一个长文档工程问题，还包括：

- 版面过滤
- 成本控制
- 状态恢复
- 大规模批量处理
- 后处理可解释性

### 为什么要拆成多阶段？

因为 OCR、诊断、修复、结构恢复的失败模式不同。拆开以后更容易 debug，也更容易知道错误发生在哪一层。

### 最难的地方是什么？

可以选一个你最熟的讲：

- 长文档 checkpoint / resume
- 页眉和图注漏回正文
- A3 双页导致版面分流不稳
- 在保正文召回和 aggressive 清洗之间做权衡

## 展示时不要这样讲

- 不要一上来就按文件名念模块
- 不要说成“我做了一个 OCR bot”
- 不要说成“准确率完美”
- 不要只讲 prompt，不讲工程设计

## 更适合的讲法

始终围绕这条主线：

1. 文档问题是什么
2. 为什么简单 OCR 不够
3. 你的系统怎么拆模块
4. 你解决了哪些具体错误
5. 最终输出为什么更适合研究/RAG
